# 移行対応 工数整理（BigQuery→Snowflake ＋ GAS/Python→dbt）

現行処理の全体像は [`現行処理整理.md`](現行処理整理.md) を参照。本書はそれを踏まえた**移行スコープ・項目・リスク・工数**の整理。

## スコープ / 前提
- **スコープ**：**mart 作成まで**（取込〜dbt marts）。mart 以降の出力（計画/支社別/目標シート生成 GAS、Gemini 提案）は対象外。
- **目的**：① BigQuery → **Snowflake** 移行、② GAS/Python の処理を可能な限り **dbt に集約**。

## 結論
- **mart までの“変換ロジック”（縦持ち化・クレンジング・表記ゆれ・型変換・集計）は dbt に集約可能。**
- ただし **“取込（スプレッドシート→ウェアハウス）”は dbt の責務外**（dbt は ELT の T 専用）。**EL 層は別途必須＝現状の TROCCO を継続**。
- 縦持ち⇄横持ちは **`UNPIVOT`/`PIVOT` で対応可能＝低リスク**。**EL層（TROCCO）も単純転送のため低リスク**。重いのは **変換移管・方言(PERCENTILE_CONT)・結果突合**。

### dbt の役割境界（なぜ取込だけ不可か）
- dbt は**ウェアハウス内の既存データを変換**するツール。スプレッドシート読取（現状 `gspread`／TROCCO）は持たない。
- BigQuery は「Sheets を外部テーブル化」して dbt から読む裏技があるが、**Snowflake は Google Sheets を直接読めない**ため、Snowflake では **EL 層が必須**。

## 現行の実体化設定（参考：移行はそのまま踏襲可）
`dbt_project.yml` のレイヤー設定のみ（モデル個別 `config()` なし）。

| レイヤー | スキーマ | materialized | 備考 |
|---|---|---|---|
| staging | `docomo_event_staging` | **view** | raw参照の軽整形 |
| intermediate | `docomo_event_intermediate` | **table** | 毎回フル洗い替え |
| marts | `docomo_event_mart` | **table** | 毎回フル洗い替え（最大 fact 約174万行） |

- `+persist_docs: {relation, columns}` … description をテーブル/カラムCOMMENTへ反映（dbt-snowflake対応）。
- **incremental 不使用**（全 table が `CREATE OR REPLACE` 洗い替え）。
- Snowflake でも view/table はそのまま使えるため**実体化はリスク低**。ただし毎回フル再作成のため、**コスト最適化したい場合のみ** marts/重い中間の incremental 化を将来検討。

## リスクが集中する2点
- **EL層（A）は単純転送のため低リスク**（TROCCO継続が素直）。残る注意点は以下2点に集約。
1. **C2/E1：`PERCENTILE_CONT` の窓形→集計形 書換＋数値突合** — 方言差で結果がズレ得る唯一の計算ロジック。
2. **D3/E1：データ品質と結果一致** — 重複・孤児・マスタ未登録の扱いを移行時に確定し、BQ版と数値突合。

## To-Be 構成（Snowflake・〜mart）
```
スプレッドシート ──[EL: TROCCO/Fivetran/dlt]──▶ Snowflake raw（横持ち＋マスタを忠実に着地）
                         │  すべて dbt
                         ▼
        staging（UNPIVOTで縦持ち化＋クレンジング＋型変換）
        → intermediate（表記ゆれ解消・デシル・Zスコア・目標値）
        → marts（fact_facility_performance_slots）
```
- dbt が担う範囲が **raw着地直後〜mart** まで拡大（GAS/Python の変換責務を集約）。
- EL 層（TROCCO）のみ dbt 外。
